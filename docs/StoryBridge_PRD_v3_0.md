| 항목 | 내용 |

| **슬로건** | "우리 아이의 이야기, 함께 만들어요" |
| **버전** | MVP v3.0 |

### 핵심 가치 제안

- **초개인화 아바타:** 아이 사진 1장 → AI 아바타(지브리풍 / 사진닮은꼴 등) 즉시 생성
- **AI 자동 스토리 생성:** Carol Gray Social Stories™ 10.4 기준 + 설명문·조망문·지시문 구조를 AI가 자동 적용, YesChat.ai 방식의 LLM 기반 실시간 생성·인라인 편집
- **3-Track 생성 모델:** 치료사·보호자·교사가 각자의 전문 맥락에서 스토리를 직접 생성하고, 역할 기반 협업·승인 플로우로 품질 보증
- **스토리 Pool:** 연령대별·카테고리별 사전 제작 템플릿 제공 (학교생활·또래관계 / 일상생활·감정조절)

## 2. 배경 및 근거

### 문제 정의

자폐 스펙트럼 장애(ASD) 아동은 사회적 상황에서의 맥락 이해와 행동 전이에 어려움을 겪는다. 기존 소셜 스토리는 다음 세 가지 구조적 한계를 지닌다.

1. **시각적 자아 일치도 부재:** 일반화된 캐릭터 사용으로 아동의 자기 참조 효과(Self-Reference Effect) 유도 불가
2. **인지적 과부하:** 평면적 정보 나열로 작업 기억(Working Memory) 용량이 부족한 ASD 아동에게 과부하 발생
3. **제작 현실성 한계:** 보호자·치료사·교사가 개별 상황에 맞는 스토리를 매번 수작업으로 제작하는 것은 시간·비용 면에서 불가능에 가까움

### 이론적 근거

| 이론 | 적용 |
|---|---|
| 인지부하이론 (Sweller, 1988) | 시공간적 청킹 + 누적 제시로 외재적 인지부하 감소; 이전 단계 정보를 화면에 오프로드하여 작업기억 부담 경감 |
| 상황 모델 이론 (Zwaan & Radvansky, 1998) | 시간·공간 단서 명시화 및 누적 제시로 5차원 상황 모델 갱신 지원 |
| 청킹 이론 (Miller, 1956; Gobet & Simon, 1996) | 누적 제시가 청크 간 연결 고리 역할 → 절차 누락 방지, 순차적 처리 강점 활용 |
| 자기 참조 효과 (Rogers et al., 1977) | AI 아바타·맞춤 상황으로 자기 관련성 극대화 |
| 실행 기능 이론 (Miyake et al., 2000) | 청킹 기반 외부 비계로 계획 수립·억제 통제 지원 |
| 다중 자원 이론 (Wickens, 1992) | 혼합 청킹이 시각-공간 + 언어-시간 경로를 동시 활성화 → 단일 청킹 대비 EF 수행 향상 |
| 누적 제시 효과 (강현정, 2026) | 순차 제시 대비 누적 제시 방식이 절차 수행 완결률 향상; 혼합 청킹 + 누적 제시 조합이 EF 및 상황 몰입에 가장 강력한 효과 (EF M=80.64, SIS-ASD M=4.45) |

## 3. 타겟 사용자 (페르소나)

### 사용자 그룹

| 페르소나 | 연령/역할 | 주요 니즈 | 핵심 페인 포인트 |
|---|---|---|---|
| 🧒 **아동** | 7~18세 ASD 진단 | 나와 닮은 주인공, 큰 버튼, 음성 읽기 | 일반 캐릭터로 몰입 불가, 정보 과부하 |
| 👩 **보호자** | 30~50대 부모/양육자 | 빠른 스토리 생성, 전문가 소통, 최종 승인권 | 매번 수작업 제작, 치료사와 소통 단절 |
| 🩺 **치료사** | 언어·행동치료사 (SLP/ABA) | 다수 아동 스토리 직접 생성·관리, 자문 제공; **ABC 행동 관찰 기록 및 기능 분석** | 아동별 맞춤 스토리 제작 시간 과다; 관찰 데이터와 스토리 설계 간 연결 도구 부재 |
| 👩‍🏫 **교사** | 특수교사·일반교사 | 학교 맥락 스토리 직접 생성, 가정 연계 | 가정과의 스토리 연계 어려움, 개별 제작 부담 |

### 연령대별 아동 세그먼트 (5구간, v3.1 개편)

| 세그먼트 | 연령 | 특징 | UI 고려사항 |
|---|---|---|---|
| 미취학 | 5~6세 | 초간단 문장, 친숙한 일상 상황 | 최대 버튼, 그림 중심, 짧은 문장 |
| 초(저학년) | 7~9세 | 짧은 문장, 단순 상황 | 아주 큰 버튼, 그림 중심, 1문장/페이지 |
| 초(고학년) | 10~12세 | 또래 관계·감정 복잡화 | 보통 버튼, 그림+텍스트 균형 |
| 중학생 | 13~15세 | 자기 이해·또래 갈등 | 텍스트 비중 증가, 성숙한 디자인 |
| 고등학생 | 16~18세 | 학교생활·자기 이해 심화 | 텍스트 중심, 성인 감성 디자인 |

### F2. 스토리 Pool (템플릿 라이브러리)

연령대(5구간) × 카테고리(2종) 교차 구성으로 총 **10개 섹션**의 사전 제작 템플릿 제공.

각 템플릿은 아동 프로필 반영 후 AI가 즉시 커스터마이징. **보호자·치료사·교사 모두 사용 가능.**

#### 카테고리 A — 학교생활·또래관계

| 연령대 | 템플릿 예시 |
|---|---|
| 5~6세 | 선생님께 인사하기, 친구와 함께 놀기, 차례 기다리기 |
| 7~9세 | 급식 줄 서기, 쉬는 시간 놀이 참여, 선생님께 도움 요청하기 |
| 10~12세 | 발표하기, 친구에게 먼저 말 걸기, 모둠 활동 참여하기 |
| 13~15세 | 그룹 과제 의견 말하기, 갈등 상황에서 감정 표현하기 |
| 16~18세 | 수업 토론 참여, 또래 갈등 중재, 학교생활 루틴 관리 |

#### 카테고리 B — 일상생활·감정조절

| 연령대 | 템플릿 예시 |
|---|---|
| 5~6세 | 화가 날 때 멈추기, 새로운 장소 가기, 잠자리 루틴 |
| 7~9세 | 화가 날 때 멈추기, 물건 정리하기, 새로운 장소 가기 |
| 10~12세 | 실수했을 때 대처, 차례 기다리기, 루틴 변화 받아들이기 |
| 13~15세 | 스트레스 조절 전략, 사과하고 화해하기, 감정 일기 쓰기 |
| 16~18세 | 자기 조절 전략, 진로 불안 다루기, 관계 갈등 해결하기 |

### F3. AI 스토리 자동생성 — 3-Track 생성 모델

스토리 생성 주체(역할)에 따라 입력 컨텍스트와 AI 프롬프트 파이프라인을 분기한다. 세 Track 모두 Carol Gray 10.4 기준과 6WH 구조를 공통으로 적용하되, **입력 가이드·프롬프트 초점·공유 방향**이 다르다.

#### Track A — 치료사 주도 생성

**AI 프롬프트 특이사항:**
- 치료 목표 태그를 프롬프트에 자동 주입 → 목표 행동 중심 지시문 배치

#### Track B — 보호자 주도 생성

**생성 맥락:** 일상·가정 상황 기반. 아동을 가장 잘 아는 보호자가 직접 상황을 묘사하는 경우.

**입력 화면 구성:**

```
┌─────────────────────────────────────────────────┐
│  우리 아이에게 어떤 이야기가 필요한가요?              │
│  ┌─────────────────────────────────────────────┐ │
│  │                                             │ │
│  │  (자유롭게 상황을 적어주세요)                  │ │
│  │                                             │ │
│  └─────────────────────────────────────────────┘ │
│                                                  │
│  ✅ 이런 내용이 들어가면 더 좋은 이야기가 돼요         │
│  ┌─────────────────────────────────────────────┐ │
│  │ 👤 누가   누가 등장하나요? (아이, 친구, 선생님)  │ │
│  │ ⏰ 언제   언제 일어나는 일인가요? (점심시간)     │ │
│  │ 📍 어디서  어디에서 일어나나요? (급식실)         │ │
│  │ 💬 무엇을  어떤 상황인가요? (새치기를 당했어요)   │ │
│  │ ✋ 어떻게  어떻게 행동하면 좋을까요? (차분히 말해요)│ │
│  │ ❤️ 왜     왜 그렇게 하면 좋을까요? (모두 기다려요) │ │
│  └─────────────────────────────────────────────┘ │
│                                                  │
│  💡 예시처럼 적어보세요                            │
│  "민준이가(누가) 점심시간에(언제) 급식실에서(어디서)   │
│   줄을 서다 친구가 새치기를 해서 화가 나요(무엇을).    │
│   화를 참고 선생님께 차분히 말하는 법(어떻게)을        │
│   알려주고 싶어요. 모두가 차례를 기다리니까요(왜)."     │
│                                                  │
│              [ 스토리 만들기 ✨ ]                  │
└─────────────────────────────────────────────────┘
```

**AI 프롬프트 특이사항:**
- 보호자 친화적 언어로 6WH 요소 추출 (전문 용어 미사용)
- 생성 후 치료사에게 "검토 요청" 알림 발송 (선택)
- 치료사 미연결 상태에서도 단독 생성·사용 가능

**상황 유형별 예시 문구 (선택 시 입력란에 자동 삽입):**

| 상황 유형 | 예시 문구 (6WH 반영) |
|---|---|
| 또래 갈등 | "지호가(누가) 쉬는 시간에(언제) 교실에서(어디서) 친구가 장난감을 빼앗아 속상해해요(무엇을). 화내지 않고 '돌려줘'라고 말하는 법(어떻게)을 배우면 좋겠어요. 그래야 친구와 사이좋게 지낼 수 있으니까요(왜)." |
| 감정 조절 | "서연이가(누가) 수업 중에(언제) 교실에서(어디서) 답을 틀리면 부끄러워 울어버려요(무엇을). 실수해도 괜찮다고 생각하며 다시 시도하는 법(어떻게)을 알려주고 싶어요. 누구나 실수하며 배우니까요(왜)." |
| 일상 전환 | "준우가(누가) 잠자기 전(언제) 집에서(어디서) 놀이를 멈춰야 할 때 떼를 써요(무엇을). 자기 전 루틴을 자연스럽게 받아들이는 법(어떻게)을 알려주고 싶어요. 푹 자야 다음 날 기분이 좋으니까요(왜)." |
| 새로운 환경 | "하은이가(누가) 다음 주에(언제) 치과에서(어디서) 처음 진료를 받아요(무엇을). 무서워하지 않고 의사 선생님 말을 따르는 법(어떻게)을 미리 준비시키고 싶어요. 건강한 이를 지킬 수 있으니까요(왜)." |

#### AI 제목 자동 생성

- 6WH 추출 결과를 바탕으로 GPT-4o-mini가 **문장형 요약 제목**(15~25자, 아동 이름을 자연스럽게 포함)을 본문 생성과 **병렬**로 생성 (예: "민준이의 급식 줄 서기 연습 이야기")
- 스토리 상세 화면에서 제목 옆 연필 아이콘으로 **인라인 수정** 가능 — 단, 생성자 본인만 (AI 제안을 사람이 검토·확정하는 패턴과 동일선상)
- AI 생성 실패 시 `${아동 이름}의 이야기` 로 폴백

#### 공통 생성 후 편집

- **인라인 편집:** 생성된 각 페이지 텍스트 직접 수정 가능
- **AI 재생성:** 페이지 단위 "다시 생성" 버튼
- **페이지 일러스트 — 등록 아바타 기반 캐릭터 일관성:** 아동의 기본 아바타 이미지가 등록되어 있으면 이를 참조 이미지로 삼아 "동일 캐릭터를 새 장면에 배치"하는 방식(`flux-kontext-pro`)으로 생성 → 페이지마다 얼굴·헤어·옷차림이 일관되게 유지됨. 아바타 미등록 시 장면 묘사만으로 그리는 방식(`flux-schnell`)으로 생성
- **아바타 이미지 교체:** 각 페이지 이미지 변경 가능
- **청킹 전략 변경:** 치료사만 생성 후에도 청킹 유형(시간적/공간적/혼합) 및 제시 방식(누적/순차) 재설정 가능
- **삭제:** 본인이 생성했거나(creator) 본인 아동의 스토리인 경우(보호자) 삭제 가능 — 이미지 Storage 파일까지 함께 정리

#### 6WH 작성 가이드 항목 (공통)

| 아이콘 | 항목 | 가이드 질문 | 예시 |
|---|---|---|---|
| 👤 | **누가 (Who)** | 누가 등장하나요? | 민준이, 친구들, 선생님 |
| ⏰ | **언제 (When)** | 언제 일어나는 일인가요? | 점심시간 |
| 📍 | **어디서 (Where)** | 어디에서 일어나나요? | 학교 급식실 |
| 💬 | **무엇을 (What)** | 어떤 상황인가요? | 줄 서다 친구가 새치기를 해서 화가 나요 |
| ✋ | **어떻게 (How)** | 어떻게 행동하면 좋을까요? | 깊게 숨 쉬고 선생님께 차분히 말해요 |
| ❤️ | **왜 (Why)** | 왜 그렇게 하면 좋을까요? | 모두가 차례를 기다리고 있으니까요 |

> **가이드는 강제가 아닙니다.** 모든 항목을 채우지 않아도 스토리를 만들 수 있으며, 빠진 정보는 AI가 후속 질문으로 자연스럽게 보완합니다.

#### AI 생성 방식 (공통)

- **자유 줄글 → LLM 분석 → 소셜 스토리 변환** 파이프라인
- GPT-4o가 줄글에서 6WH 요소를 **자동 추출**하여 Carol Gray 10.4 기준에 맞게 재구성
- 누락된 항목이 있으면 AI가 **자연스러운 후속 질문 1~2개** 제시
- YesChat.ai 방식의 실시간 스트리밍 생성
- 아동 프로필(연령·관심사·친숙 환경)을 프롬프트에 자동 결합
- **청킹 유형 파라미터** (`chunking_type`: temporal / spatial / mixed) 프롬프트 자동 주입
- **제시 방식 파라미터** (`presentation_mode`: cumulative / sequential) 프롬프트 자동 주입
  - `cumulative` 선택 시: 각 페이지 생성 시 이전 단계 요약 단서를 하단 절차 스트립 텍스트로 함께 생성
  - `sequential` 선택 시: 현재 페이지 단계만 생성 (기존 방식)

#### 스토리 구성 제한 (MVP)

- 최대 **10페이지**
- 페이지당: 이미지(AI 생성 또는 직접 업로드) + 텍스트(설명문·조망문·지시문)
- 제목 페이지(1) + 본문 페이지(최대 8) + 마무리 페이지(1)

### F4. 스토리 뷰어

Pictello의 읽기 모드 구조를 참조하여 ASD 아동 최적화 UI로 구현.

| 모드 | 설명 |
|---|---|
| **Manual** | 탭/클릭으로 페이지 전환, 음성 재생 |
| **Auto Play** | 음성 완료 후 자동으로 다음 페이지 이동 |
| **Slideshow** | 설정된 시간(3/5/7초) 간격으로 자동 진행 |

**추가 기능:**
- TTS 한국어 음성 (Google Cloud TTS, 남성/여성/아동 선택)
- 읽히는 단어 하이라이트 동기화
- 큰 버튼 UI (최소 48px 터치 영역)
- 고대비 모드 토글 (배경 흰색, 텍스트 검정)
- 스와이프 페이지 전환 (모바일)
- 뷰어 상단 생성 주체 표시 (예: 🩺 치료사 생성 / 👩‍🏫 교사 생성)
- Track C 스토리의 경우 가정 연계 메모 하단 표시 (보호자·치료사 전용)
- 홈 버튼으로 라이브러리 복귀

#### 누적 제시(Cumulative) 뷰어 UI

`presentation_mode = cumulative`로 생성된 스토리는 각 페이지 하단에 **절차 진행 스트립(Cumulative Strip)** 을 표시한다. 이전 단계의 시각적 단서가 사라지지 않고 축적되어 남아, 아동이 현재 단계의 맥락을 유지하며 전체 절차를 누락 없이 수행하도록 돕는다.

```
┌──────────────────────────────────────┐
│  🖼️ [현재 페이지 아바타 이미지]        │
│                                      │
│  "나는 줄 맨 뒤에 서서               │
│   차례를 기다릴 수 있어요."            │
│                                      │
│  ── 지금까지 한 것들 ──               │
│  ✅ 먼저, 급식실 앞에 도착했어요       │
│  ✅ 그 다음, 친구들이 있는지 봤어요    │
│  ▶ 지금, 줄 맨 뒤에 서서 기다려요     │
└──────────────────────────────────────┘
```

**절차 진행 스트립 규칙:**
- 완료 단계: ✅ 아이콘 + 흐린 텍스트 (Soft Mint `#A8D8D8`)
- 현재 단계: ▶ 아이콘 + 진한 텍스트 + 강조 테두리 (Coral Accent `#F08060`)
- 미래 단계: 표시하지 않음 (인지 부하 최소화)
- 스트립 영역 최대 높이: 화면의 30% 이내 (스크롤 가능)

**순차 제시(Sequential) 뷰어 UI:**
`presentation_mode = sequential`인 경우 절차 진행 스트립 미표시. 현재 페이지 내용만 표시 (기존 방식).

### F5. 협업·승인 플로우

#### 그룹 구조

```
보호자 (최종 승인권 보유 + 초대 코드 발급)
├── 치료사 (Track A 생성 + 수정 제안 권한)
└── 교사   (Track C 생성 + 수정 제안 권한)
```

- 아동 프로필을 등록하면 **그룹이 자동 생성**됨 (보호자가 별도로 "그룹 만들기"를 하지 않음)
- 초대 코드는 "아이 프로필" 화면에서 보호자가 확인·복사하여 치료사·교사에게 전달
- 치료사·교사는 "협업 공간"에서 초대 코드를 입력해 그룹에 참여 (코드를 아는 것 자체가 참여 권한의 근거)
- 그룹으로 연결되면 해당 아동의 프로필·스토리·행동 관찰 기록을 보호자와 동일하게 열람 가능 (전역 "내 아이 목록"에 함께 표시되며 "🔗 연결된 아이"로 구분 표시)
- 그룹 내 모든 스토리 공유 및 자문 가능

#### Track별 협업 시퀀스

**Track A — 치료사 주도**

```
[치료사]
치료 목표 기반 스토리 생성·편집
        ↓
"가정 활용 요청" 알림 → [보호자]
        ↓
보호자 열람 및 확인
        ↓
[확인 완료] → 가정에서 아동과 함께 읽기 시작
[수정 요청] → 피드백 댓글 → 치료사 재수정
```

**Track B — 보호자 주도**

```
[보호자]
일상 상황 기반 스토리 생성·편집
        ↓
"치료사 검토 요청" 발송 (선택)
        ↓
[치료사] 수정 사항 작성 후 "승인 요청" 전송
        ↓
[보호자] 변경 전/후 비교 화면 확인
        ↓
[승인] → 스토리 즉시 반영 + 치료사에게 완료 알림
[거절] → 원본 유지 + 피드백 댓글
```

**Track C — 교사 주도**

```
[교사]
학교 맥락 기반 스토리 생성·편집
        ↓
그룹 전체(보호자·치료사) 자동 공유 알림
        ↓
보호자·치료사 열람 및 자문 댓글 작성
        ↓
[보호자] 가정 연계 확인 완료 표시
[치료사] 치료 목표와 정합성 검토 → 수정 제안 가능
        ↓
교사가 수정 제안 수락 시 → 보호자 최종 승인
```

#### 자문 댓글 구조

- **스토리 레벨:** 스토리 전체에 대한 의견
- **페이지 레벨:** 특정 페이지 옆 말풍선 아이콘 탭 → 해당 페이지 댓글 작성
- 댓글 작성자 표시 (이름 + 역할 배지)
- 실시간 읽음 확인 (✓ / ✓✓)
- 알림: 앱 내 배지 + 이메일 알림
- 승인 히스토리 영구 보관

#### 수정 제안·승인 플로우 (구현 상세) — NEW v3.6

- **페이지별 인라인 에디터** — 스토리 상세 화면에서 각 페이지(설명문/조망문/지시문)를 연필 아이콘으로 바로 펼쳐 수정
  - **보호자 / 본인이 생성한 스토리(creator)**: "저장" 즉시 반영 (승인 절차 없음)
  - **그 외 역할(치료사·교사 등 creator가 아닌 경우)**: "수정 제안 보내기" — **제안 사유**(선택) 입력 후 보호자에게 승인 요청 전송. 아이(child) 역할은 제안 불가
- **승인 요청 카드** — 어느 스토리의 몇 페이지에 대한 제안인지 표지(스토리 제목 · N페이지)로 표시, 변경 전/후를 단어 단위로 비교해 **삭제된 단어는 빨간 취소선**, **추가된 단어는 노란 하이라이트**로 표시, 제안 사유 함께 표시, 승인/거절 버튼
- **승인 내역(history) 탭** — 처리 완료(승인/거절)된 제안을 그룹의 스토리 기준으로 모아 보여줌 (동일한 표지·하이라이트 표시)
- **알림 연동** — 제안 전송 시 보호자에게 "수정 제안" 알림(approval_request) + 제안한 본인에게도 "제안을 보냈어요(보호자 승인 대기중)" 확인 알림(approval_sent); 보호자 승인/거절 시 제안자에게 결과 알림(approval_result). 모든 알림 본문에 변경 필드명(설명문/조망문/지시문)과 변경 내용 일부를 한 줄로 요약해 포함

|---|---|---|
| **S (Sensory)** | 감각 자기자극 | 특정 감각 입력에 반애 또는 취하려는 행동 |
| **E (Escape/Avoid)** | 도피·회피 | 어려운 과제나 상황에서 벗어나려는 행동 |
| **A (Attention)** | 관심 요청 | 특정인의 주의를 얻으려는 행동 |
| **T (Tangible)** | 유형적 요구 | 특정 물건이나 활동을 얻으려는 행동 |

- 복수 항목 선택 가능 (예: S + E 동시 선택)
- **AI 자동 분석 (GPT-4o):** ABC 텍스트를 기반으로 ① SEAT 기능 분류(복수 가능, 신뢰도·근거 포함), ② **대체행동 목표 초안**까지 함께 제안
  - 대체행동 제안은 분류된 SEAT 기능과 동일한 욕구를 더 적응적인 방식으로 충족하도록, 아이의 현재 수준에서 바로 가르칠 수 있게 구체적으로 작성
  - "⭐ AI 추천" 표시와 함께 제시되며, **치료사가 검토 후 "이 제안 적용하기"로 직접 입력란에 반영**하는 초안-검증 구조 (AI가 대신 결정하지 않음)

#### ABC → Track A 연결 플로우

```
[치료사] ABC 기록 저장 완료 (+ 선택적으로 AI SEAT/대체행동 제안 검토 후 적용)
        ↓
[이 관찰로 Track A 스토리 만들기] 버튼
        ↓
 Track A 입력창에 raw_input 자동 채움 (다시 입력할 필요 없음)
   [관찰된 도전 상황] → A(선행자극)
   [문제 행동]        → B(행동)
   [현재 결과]        → C(결과)
   [대체행동 목표]     → replacement_behavior (있을 때만 포함)
   - SEAT 분류 → 치료 목표 태그(therapy_goal_tags) 자동 매핑 힌트
        ↓
 Track A 스토리 생성 완료 후 observation_id 링크 저장
```

#### 관찰 히스토리

- 아동별 시계열 로그 (관찰 일시, 환경, SEAT 분류 표시)
- 기능 분포 요약: SEAT 설정 열 목구막 차트 (텍스트 기반)
- 스토리와 연결된 관찰은 해당 스토리 상세에서 표시

## 7. 문장 유형 구조

Carol Gray 소셜 스토리의 핵심 3가지 문장 유형을 각 페이지에 자동 적용.

### 문장 유형 정의

| 유형 | 정의 | 예시 |
|---|---|---|
| **설명문 (Descriptive)** | 상황에 대한 객관적 사실을 서술 | "급식 시간에 친구들은 줄을 서서 밥을 받아요." |
| **조망문 (Perspective)** | 타인의 감정·생각·의도를 묘사 | "선생님은 모든 친구들이 공평하게 기다려 주기를 원해요." |
| **지시문 (Coaching)** | 아동이 할 수 있는 대처 행동 제시 | "나는 숨을 한 번 크게 쉬고 차례를 기다릴 수 있어요." |

### 페이지별 구성 예시

```
[페이지 3 예시 — 급식 줄 서기]

🖼️ [아바타 이미지: 민준이가 줄을 서는 모습]

📌 설명문
"점심시간에 급식실에서 친구들은 차례로 줄을 서서 밥을 받아요."

💭 조망문
"선생님과 친구들은 모두가 기다리면 더 빨리 밥을 먹을 수 있다고 생각해요."

✅ 지시문
"나는 줄 맨 뒤에 서서 내 차례를 기다릴 수 있어요."
```

### AI 프롬프트 비율 규칙

```
설명문 + 조망문 : 지시문 = 최소 4 : 1
- 10페이지 기준: 지시문 포함 페이지는 최대 2페이지
- 나머지 페이지는 설명문 또는 조망문 중심
- 전체 문장의 최소 50% 는 긍정·칭찬 내용
```

## 8. 시공간적 청킹 전략

인지부하이론(Sweller, 1988), 상황 모델 이론(Zwaan & Radvansky, 1998), 다중 자원 이론(Wickens, 1992)에 근거한 정보 설계 원칙. 청킹 전략은 **유형(Type)** 과 **제시 방식(Presentation Mode)** 두 차원으로 구성된다.

### 차원 1 — 청킹 유형 (3가지)

청킹 유형은 **하나의 사회적 상황을 어떤 기준으로 분절하느냐**를 결정한다.

#### 1) 시간적 청킹 (Temporal Chunking)

사건을 **시간 순서**에 따라 분절. 표지어가 시간의 흐름을 명시한다.

| 항목 | 내용 |
|---|---|
| **표지어** | 먼저 → 그 다음 → 마지막으로 |
| **지원 EF** | 계획 수립(Planning), 순서화 |
| **상황 모델 차원** | 시간(Time) |
| **권장 대상** | 루틴 형성 어려움, 단계 순서를 자주 건너뛰는 아동 |
| **연구 EF 점수** | M = 61.88 (강현정, 2026) |

**페이지 예시 — 급식 줄 서기:**
```
페이지 1: "먼저, 급식실 문 앞에 도착하면 친구들이 있는지 확인해요."
페이지 2: "그 다음, 줄의 맨 뒤에 가서 서요."
페이지 3: "그 다음, 두 손을 모으고 내 차례를 기다려요."
페이지 4: "마지막으로, 내 차례가 되면 차분하게 밥을 받아요."
```

#### 2) 공간적 청킹 (Spatial Chunking)

사건을 **장소·환경의 변화**에 따라 분절. 표지어가 공간 맥락을 명시한다.

| 항목 | 내용 |
|---|---|
| **표지어** | 교실에서 → 복도에서 → 급식실에서 |
| **지원 EF** | 환경 전환 대처, 억제 통제 |
| **상황 모델 차원** | 공간(Space) |
| **권장 대상** | 환경 전환(Transition) 불안이 심한 아동, 장소별 행동 규칙 혼란 아동 |
| **연구 EF 점수** | M = 65.23 (강현정, 2026) |

**페이지 예시 — 급식 줄 서기:**
```
페이지 1: "교실에서, 선생님이 '급식 시간'이라고 말씀하시면 가방을 제자리에 두어요."
페이지 2: "복도에서, 친구들과 나란히 조용히 걸어가요."
페이지 3: "급식실 앞에서, 줄에 차례대로 서요."
페이지 4: "급식실 안에서, 번호 순서대로 밥을 받아요."
```

#### 3) 혼합 청킹 (Mixed Chunking) ⭐ 디폴트

시간 표지어 + 공간 표지어를 **동시에** 사용. 한 페이지에 "어디서, 언제(순서)"가 모두 명시된다.

| 항목 | 내용 |
|---|---|
| **표지어** | 교실에서, 먼저 → 복도에서, 그 다음 → 급식실에서, 마지막으로 |
| **지원 EF** | 계획 수립 + 환경 전환 + 억제 통제 동시 지원 |
| **상황 모델 차원** | 시간(Time) + 공간(Space) 동시 |
| **권장 대상** | 대부분의 ASD 아동 / 복합 목표 / 실행 기능 전반 향상 목표 |
| **연구 EF 점수** | M = 74.68 (강현정, 2026) — **6조건 중 최고** |

**이론적 근거:** Wickens(1992) 다중 자원 이론에 따라 시각-공간 처리 경로와 언어-시간 처리 경로는 독립적으로 작동하므로 두 경로를 동시 활성화하는 혼합 청킹이 단일 청킹보다 더 높은 EF 수행을 유도한다.

**페이지 예시 — 급식 줄 서기:**
```
페이지 1: "교실에서, 먼저 선생님 말씀이 끝나면 가방을 두어요."
페이지 2: "복도에서, 그 다음 친구들 옆에서 조용히 걸어요."
페이지 3: "급식실 앞에서, 그 다음 줄 맨 뒤에 서서 기다려요."
페이지 4: "급식실 안에서, 마지막으로 차분하게 밥을 받아요. 잘했어요! 🎉"
```

### 차원 2 — 제시 방식 (2가지)

제시 방식은 **정보를 어떻게 화면에 보여주느냐**를 결정한다. 청킹 유형과 독립된 별도 차원이다.

#### 1) 누적 제시 (Cumulative Presentation) ⭐ 디폴트

이전 단계의 시각적 단서를 화면에 **누적하여 남기는** 방식. 페이지를 넘겨도 완료 단계가 절차 진행 스트립에 축적된다.

```
[순차 제시]                     [누적 제시]
페이지 3: 단계③만 표시           페이지 3: 단계①✅ + 단계②✅ + 단계③▶ 표시
→ 이전 맥락 사라짐                → 전체 절차 맥락 화면에 유지
```

**인지공학적 근거:** 이전 단계 정보를 화면에 오프로드하여 작업기억의 부담을 경감한다(Sweller, 1988). 아동이 "지금 몇 번째 단계인지", "무엇을 완료했는지"를 내부 기억에 의존하지 않아도 된다.

**연구 근거:** 강현정(2026) 프로포절에서 누적 제시 방식이 순차 제시 대비 절차 수행 완결률 향상에 기여함을 확인. 혼합 청킹 + 누적 제시 조합에서 EF M=80.64, SIS-ASD M=4.45로 6조건 중 최고 결과.

#### 2) 순차 제시 (Sequential Presentation)

현재 페이지 내용만 표시하는 **기존 방식**. 이전 단계 정보는 화면에서 사라진다.

**권장 상황:** 치료사가 임상적 판단에 따라 특정 아동에게 한 번에 하나의 자극만 제시해야 할 때.

### 청킹 전략 2차원 조합 비교

| | 시간적 | 공간적 | 혼합 ⭐ |
|---|---|---|---|
| **순차 제시** | 기본 가능 | 기본 가능 | 가능 |
| **누적 제시 ⭐** | 가능 | 가능 | **최강 조합** |

> **디폴트 설정:** 혼합 청킹 + 누적 제시
> **수정 권한:** 치료사만 변경 가능. 보호자·교사는 설정값 열람만 가능.
> 모든 생성 화면에서 선택 UI 하단에 고정 표시:
> **"'혼합 청킹 + 누적 제시' → 연구기반 가장 강력한 조합으로 추천드립니다."**

## 10. 협업·승인 플로우

### 공통 규칙

- 모든 스토리의 **최종 승인권은 보호자**에게 있음
- 치료사·교사가 **본인이 생성한 스토리를 수정**하는 경우 → 보호자 승인 불필요
- 치료사·교사가 **타인(보호자 포함) 생성 스토리를 수정** 제안하는 경우 → 보호자 승인 필요
- 모든 수정 이력은 **승인 히스토리**에 영구 보관
- 생성 주체(역할·Track)는 스토리 메타데이터에 기록되어 뷰어 상단에 표시

### 상세 시퀀스

Track별 시퀀스는 F5 협업·승인 플로우 섹션 참조.

```
공통 단계 요약:
1. 생성 주체(치료사/보호자/교사)가 스토리 완성 후 저장
      ↓
2. Track에 따라 자동 알림 발송 (알림 매트릭스 참조)
      ↓
3. 수신자 열람 → 자문 댓글 또는 수정 제안
      ↓
4-A. [승인] → 스토리 즉시 반영 + 요청자에게 완료 알림
4-B. [거절] → 원본 유지 + 피드백 댓글 → 재수정 가능
      ↓
5. 전체 이력 "승인 히스토리" 탭에 영구 기록
```

### 알림 매트릭스

| 이벤트 | 보호자 | 치료사 | 교사 |
|---|---|---|---|
| Track A 스토리 생성 완료 | 🔔 가정 활용 요청 | — | — |
| Track B 검토 요청 발송 | — | 🔔 검토 요청 | — |
| Track C 스토리 공유 | 🔔 새 스토리 | 🔔 새 스토리 | — |
| 수정 제안 전송 | 🔔 승인 요청 | — | — |
| 보호자 승인 완료 | — | 🔔 완료 | 🔔 완료 |
| 보호자 거절 | — | 🔔 거절+댓글 | 🔔 거절+댓글 |

> **v3.6 구현 현황** — 위 표 중 "수정 제안 전송 / 보호자 승인 완료 / 보호자 거절" 3개 행은 `notifications` 테이블 기반으로 실제 구현 완료. Track A/B/C **스토리 생성 시점**의 자동 알림(가정 활용 요청·검토 요청·새 스토리 공유)은 아직 미구현 — v4 이후 과제로 이월.
>
> **실제 구현된 알림 종류 (`notifications.type`)**
>
> | 타입 | 수신자 | 트리거 | 본문 |
> |---|---|---|---|
> | `approval_request` | 보호자 | 수정 제안 전송 | "{이름}님이 스토리 수정을 제안했어요: {제안 사유}" |
> | `approval_sent` | 제안한 본인 | 수정 제안 전송 (본인 확인용) | "{변경 요약} (보호자 승인 대기중)" |
> | `approval_result` | 제안자 | 보호자 승인/거절 | 승인: "{변경 요약} → 반영됐어요" / 거절: 보호자가 입력한 피드백 또는 "{변경 요약} 제안이 거절됐어요" |
>
> 모든 알림은 사이드바·하단 네비 배지 + `/notifications` 알림함에 표시되며, Supabase Realtime으로 즉시 수신된다.
